## Что это

**AgentData.pro** — это не "ClickUp с AI" и не "Obsidian с агентами". Это новая категория продукта: **многодоменный агентный workspace**.

Суть: AI-агенты работают автономно 24/7 в изолированных доменах знаний (здоровье, финансы, работа, фитнес), а пользователь взаимодействует с ними через Telegram и Web UI — делегирует задачи, получает инсайты, управляет исключениями.

Агенты не просто отвечают на вопросы — они проводят ресерч ночью, дебатируют между собой, накапливают знания в MD-файлах с полной историей версий, и проактивно уведомляют пользователя о важном.

**Ключевое отличие от существующих решений:**

| Продукт | Что делает | Чего не хватает |
|---------|-----------|-----------------|
| ChatGPT / Claude | Диалог по запросу | Нет автономности, нет доменов, нет памяти между сессиями |
| Notion AI / ClickUp AI | AI-помощник внутри task manager | Нет мультиагентности, нет изоляции доменов |
| Obsidian + plugins | Персональная база знаний | Нет агентов, нет коллаборации, нет real-time |
| AutoGPT / CrewAI | Мультиагентные фреймворки | Нет продуктовой оболочки, нет UI, нет персистентности |
| **AgentData.pro** | **Всё вместе** | **Домены + агенты + коллаборация + Telegram + Web UI** |

## Для кого

### Начальная аудитория
- Автор (Petrovich) + 10–15 друзей и коллег
- Технически грамотные пользователи, которым нужен персональный AI-ассистент с глубокой специализацией по доменам

### Целевая аудитория при росте
- До 70+ пользователей
- Каждый пользователь имеет **личные домены** (здоровье, финансы, фитнес) и может участвовать в **совместных доменах** (рабочие проекты, командные задачи)

### Модель использования
- **Персональное:** один человек — несколько доменов, агенты работают только на него
- **Коллаборативное:** несколько человек в одном домене, агенты видят общий контекст, задачи распределяются между участниками

## Ключевые концепции

### 1. Домены знаний (Domains)

Домен — изолированная сфера деятельности с собственными данными, агентами и памятью.

```
┌─────────────────────────────────────────────────────────┐
│                     Пользователь                        │
│                                                         │
│  ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌────────┐  │
│  │ Здоровье  │ │ Финансы/  │ │  Работа:  │ │ Фитнес │  │
│  │           │ │  Крипто   │ │ Проект X  │ │        │  │
│  │ owner     │ │ owner     │ │ member    │ │ owner  │  │
│  │ 2 agents  │ │ 1 agent   │ │ 3 agents  │ │ 1 agent│  │
│  └───────────┘ └───────────┘ └───────────┘ └────────┘  │
│       ▲               ▲             ▲            ▲      │
│       │               │             │            │      │
│       └───── PostgreSQL RLS: физическая изоляция ┘      │
└─────────────────────────────────────────────────────────┘
```

**Принципы доменов:**

- Домен имеет **owner** (создатель) и **members** (приглашённые)
- Данные одного домена **физически недоступны** из другого — PostgreSQL Row-Level Security (RLS)
- Агент получает контекст **только своего домена** — никогда не видит данные из чужих доменов
- Домены можно расшарить — пригласить друга или коллегу с ролью `member` или `viewer`

**Примеры доменов:**

| Домен | Тип | Описание |
|-------|-----|----------|
| Здоровье | Личный | Препараты, вес, анализы, рекомендации |
| Финансы/Крипто | Личный | Портфель, алерты, аналитика рынка |
| Работа: Проект X | Совместный | Задачи, координация, агенты-аналитики |
| Фитнес | Личный | Тренировки, телеметрия, прогресс |
| Личный ассистент | Личный | Календарь, маршруты, напоминания |
| Идеи/Бизнес | Личный | Исследования, бизнес-планы, заметки |

### 2. Мультиагенты (Agents)

Агент — автономная AI-сущность, привязанная к домену, с определённой ролью и набором инструментов.

**Шаблоны + кастомизация:**
- Готовые роли: аналитик, критик, ассистент, исследователь, синтезатор
- Пользователь может создать собственного агента с уникальным промптом и набором tools

**Три режима работы:**

| Режим | Триггер | Пример |
|-------|---------|--------|
| По запросу | Пользователь задаёт вопрос | "Проанализируй мои анализы крови" |
| По расписанию (cron) | Время | Каждое утро в 07:00 — напоминание о препаратах |
| Реактивный | Событие | Цена BTC упала на 10% → алерт |

**Изоляция:**
- Агент принадлежит домену и видит **только данные этого домена**
- Даже если один пользователь имеет 5 доменов — агент из домена "Здоровье" не имеет доступа к данным домена "Финансы"

### 3. MD-архитектура (Архитектура памяти)

MD-файлы — это **внутренний формат агентов**, а не UI пользователя.

```
domains/
  health/
    memory/
      supplements_protocol.md      ← агент хранит протокол приёма
      weight_analysis_2026_03.md   ← ежемесячный анализ веса
      blood_test_review.md         ← разбор анализов
    debates/
      weekly_health_review_w12.md  ← лог дебатов агентов
    research/
      magnesium_forms_comparison.md ← ресерч по запросу
```

**Почему MD:**
- **Git-versioned** — полная история знаний, diff любого изменения
- **Человекочитаемо** — для debug можно открыть в любом редакторе
- **Агентонативно** — LLM естественно генерируют и парсят Markdown

**Пользователь видит данные через Telegram/Web UI**, а не через файлы. MD — это "мозг" агента, не интерфейс человека.

### 4. Модель взаимодействия

**Telegram + Web UI** — равноправные каналы.

| Канал | Для чего |
|-------|----------|
| Telegram | Быстрые действия, уведомления, голосовые команды, мобильный доступ |
| Web UI | Глубокая работа, дашборды, управление доменами, просмотр дебатов, настройка агентов |

Оба канала подключены к одному бэкенду, одним данным, одному потоку событий. Сообщение в Telegram мгновенно отражается в Web UI и наоборот.

## Сценарий дня

> День Пети — пользователя AgentData.pro

### Утро (07:00–08:00)

**Telegram** — агент домена "Здоровье" присылает утреннее сообщение:

```
🏥 Доброе утро!

Препараты на сегодня:
• Магний цитрат — 400 мг (с завтраком)
• Витамин D3 — 5000 IU
• Омега-3 — 2 капсулы

Введи вес: (просто число, например 87.3)
```

Петя отвечает: `87.1` — агент фиксирует, обновляет тренд, молча записывает в домен.

**Тем временем** крипто-агент из домена "Финансы" алертит:

```
📊 BTC вырос на 8% за ночь ($71,200).
RSI на 4h = 78 (перекупленность).

Рекомендация: рассмотреть частичную фиксацию прибыли.
Подробный анализ: /debate crypto weekly
```

### Дорога на работу (08:30–09:00)

**Смартфон (Web UI)** — Петя открывает раздел "Ночные ресерчи":

- Агент-аналитик провёл сравнительный анализ форм магния (цитрат vs бисглицинат)
- Агент-критик нашёл слабые места в аргументации
- Синтезатор выдал итог: "Бисглицинат лучше для сна, цитрат — для мышц. Рекомендация: оставить цитрат утром, добавить бисглицинат вечером."

Петя нажимает "В работу" → задача появляется в домене "Здоровье".

### Работа (09:00–18:00)

В совместном домене "Работа: Проект X" (3 участника):

- Петя делегирует рутинную задачу агенту: "Собери метрики за неделю и подготовь отчёт"
- Агент начинает работу, прогресс виден в Web UI
- Коллега Маша видит в доменном чате: "Агент подготовил отчёт, проверьте"
- Петя замечает, что агент неправильно интерпретировал одну метрику → human-in-the-loop: корректирует и запускает повторно

### Вечер (19:00–21:00)

- Личный ассистент предупреждает о встрече завтра, предлагает маршрут с учётом пробок
- Фитнес-агент анализирует сегодняшнюю тренировку: "Объём нагрузки на 12% выше прошлой недели. Рекомендую день отдыха завтра."

### Ночь (00:00–06:00)

Агенты работают автономно:
- Крипто-агент мониторит рынок и готовит утренний брифинг
- Здоровье-агент анализирует тренд веса за месяц
- Рабочий агент делает глубокий ресерч по теме, которую Петя запросил вчера

Всё это будет готово к утру.

```

                    │                                          │
                    │  TaskIQ workers (agent orchestration)    │
                    │  APScheduler (cron-задачи агентов)       │
                    │                                          │
                    │  PostgreSQL 16                            │
                    │    (JSONB + pgvector + RLS + LTREE)      │
                    │                                          │
                    │  Redis 7                                  │
                    │    (task broker + pub/sub + Streams)     │
                    │                                          │
                    │  Netdata + Uptime Kuma → Telegram alerts │
                    │                                          │
                    └──────────────┬───────────────────────────┘
                                   │
                         WireGuard tunnel
                                   │
                    ┌──────────────┴───────────────────────────┐
                    │        Home Server (GPU)                  │
                    │        Ryzen 7 7700 / 64 GB RAM          │
                    │        RTX 5070 Ti 16 GB VRAM            │
                    │                                          │
                    │  Ollama                                   │
                    │    ├─ Qwen 2.5 14B Q4_K_M (~9 GB VRAM)  │
                    │    └─ Llama 3.1 8B Q4_K_M (~5 GB VRAM)  │
                    │                                          │
                    │  OpenAI-compatible API                    │
                    │    /v1/chat/completions                   │
                    └──────────────────────────────────────────┘
```

### LLM Routing

Кастомный роутер (~50 строк Python) с простой логикой:

```
Запрос от агента
    │
    ├─ Рутина (80% задач)
    │   └─► Ollama (локальная модель, бесплатно)
    │       ├─ Qwen 2.5 14B — основная модель
    │       └─ Llama 3.1 8B — быстрые задачи
    │
    ├─ Сложные задачи (20%)
    │   └─► OpenAI API (платно)
    │
    └─ Fallback
        └─ Домашний сервер недоступен → OpenAI API
```

Ollama уже предоставляет OpenAI-совместимый API (`/v1/chat/completions`), поэтому переключение между локальной моделью и OpenAI — это смена одного URL.

## Схема данных

### Концептуальная ER-модель

```
┌──────────┐     ┌───────────────┐     ┌──────────┐
│  users   │────►│ domain_members│◄────│ domains  │
│          │     │  (role: owner │     │          │
│ id       │     │   member      │     │ id       │
│ email    │     │   viewer)     │     │ name     │
│ password │     └───────────────┘     │ type     │
│ profile  │                           └─────┬────┘
└──────────┘                                 │
                                    domain_id│(RLS)
                           ┌─────────────────┼─────────────────┐
                           │                 │                 │
                    ┌──────┴──┐      ┌──────┴──┐      ┌──────┴──┐
                    │ agents  │      │  tasks  │      │ messages│
                    │         │      │         │      │         │
                    │ role    │      │ ltree   │      │ channel │
                    │ prompt  │      │ status  │      │ (tg/web)│
                    │ tools   │      │ assignee│      │ content │
                    │ schedule│      │ meta    │      └─────────┘
                    └────┬────┘      │ (JSONB) │
                         │           └─────────┘
                         │
              ┌──────────┼──────────┐
              │          │          │
       ┌──────┴──┐ ┌────┴────┐ ┌──┴───────┐
       │ debates │ │ memory  │ │  events  │
       │         │ │         │ │          │
       │ rounds  │ │ text    │ │ actor    │
       │ thesis  │ │ vector  │ │ action   │
       │ anti-   │ │(pgvec)  │ │ resource │
       │ thesis  │ │         │ │ domain   │
       │ synth   │ │         │ │ decision │
       └─────────┘ └─────────┘ └──────────┘
```

### Основные сущности

| Таблица | Назначение |
|---------|-----------|
| `users` | Аутентификация, профиль |
| `domains` | Домены знаний (health, work, fitness...) |
| `domain_members` | Кто имеет доступ к какому домену (owner, member, viewer) |
| `agents` | Определения агентов (роль, промпт, tools, расписание) |
| `tasks` | Рабочие элементы — задачи людей и агентов в одном потоке |
| `debates` | Сессии дебатов агентов (раунды thesis/antithesis/synthesis) |
| `events` | Append-only audit log — каждое действие человека или агента |
| `memory` | Долгосрочная память агентов (текст + pgvector embeddings) |
| `messages` | Сообщения чата (Telegram + Web, scoped to domain) |
| `files` | Метаданные MD-файлов (содержимое на файловой системе, git-versioned) |

### Ключевые паттерны

- **Все таблицы содержат `domain_id`** — RLS гарантирует изоляцию на уровне SQL-запросов
- **`events`** партиционирована по месяцам (pg_partman) — append-only, никогда не удаляется
- **LTREE** для иерархии задач: `domain.stream.task.subtask`
- **JSONB** для гибкой конфигурации агентов и метаданных задач — schema evolution без миграций

## Безопасность

### 1. Изоляция доменов

PostgreSQL Row-Level Security (RLS) — запросы **физически не могут** вернуть данные из чужих доменов. Это не фильтр на уровне приложения, а ограничение на уровне СУБД.

```sql
-- Упрощённый пример RLS policy
CREATE POLICY domain_isolation ON tasks
  USING (domain_id = current_setting('app.current_domain_id')::uuid);
```

### 2. Изоляция агентных сессий

Каждая операция агента получает **чистый контекст**, ограниченный его доменом. Агент не может запросить данные из другого домена даже при prompt injection — ограничение на уровне database session.

### 3. Аутентификация

| Компонент | Решение |
|-----------|---------|
| Хеширование паролей | Argon2id (winner PHC, memory-hard) |
| Access Token | JWT, время жизни 15 минут |
| Refresh Token | 7 дней, httpOnly cookie, SameSite=Strict |
| Хранение токенов | httpOnly cookies — недоступны из JavaScript |

### 4. Сетевая безопасность

| Уровень | Мера |
|---------|------|
| Открытые порты | Только 80, 443, 51820 (WireGuard) |
| SSH | Только через VPN |
| DDoS | Cloudflare proxy скрывает реальный IP |
| Rate limiting | Nginx `limit_req` на API и login endpoints |
| Firewall | ufw + fail2ban |

### 5. Операционная безопасность

| Аспект | Подход |
|--------|--------|
| Секреты | Только environment variables, никогда в коде |
| Бэкапы | pg_dump ежедневно, rclone в облачное хранилище, ежемесячный тест восстановления |
| Аудит | Каждое действие логируется в `events` с actor, resource, domain, decision |

## Архитектура агентных дебатов

Дебаты — механизм, при котором несколько агентов с разными ролями последовательно анализируют вопрос, чтобы дать более качественный ответ.

```
Запрос пользователя
        │
        ▼
  Orchestrator (TaskIQ)
        │
        ├──── Раунд 1: THESIS
        │     Agent A (роль: аналитик)
        │     Генерирует первичный ответ с аргументами
        │
        ├──── Раунд 2: ANTITHESIS
        │     Agent B (роль: критик)
        │     Оспаривает, находит слабые места, предлагает альтернативы
        │
        ├──── Раунд 3: SYNTHESIS
        │     Agent C (роль: синтезатор)
        │     Объединяет лучшее из обоих позиций
        │
        └──── Результат:
              ├─ Краткий итог → Telegram / Web notification
              ├─ Полный лог дебатов → MD-файл (git-versioned)
              └─ Событие → audit log (таблица events)
```

**Экономика дебатов:**
- Фиксированная стоимость: **3 LLM-вызова на раунд**
- Дополнительные раунды — только если качество недостаточно (решает orchestrator)
- При использовании локальной модели через Ollama — стоимость = 0

**Пользовательский опыт:**
- Пользователь видит **краткий итог** (1–3 предложения)
- По клику — **полный лог дебатов** с позицией каждого агента
- Каждый агент работает **в контексте своего домена** — критик тоже видит только данные домена

## Real-time архитектура

### WebSocket: сервер → клиент

```
FastAPI ──── WebSocket ────► React (TanStack Query cache invalidation)
   │
   ├─ agent.started       — агент начал работу
   ├─ agent.progress      — промежуточный прогресс
   ├─ agent.completed     — агент завершил задачу
   ├─ debate.round        — обновление раунда дебатов
   ├─ message.new         — новое сообщение в чате
   └─ system.notification — системное уведомление
```

### Reconnection и гарантия доставки

```
Клиент хранит last_event_id
        │
        ▼
Разрыв соединения → переподключение
        │
        ▼
Сервер: replay пропущенных событий из Redis Streams (TTL 24 часа)
        │
        ▼
Клиент получает все события, которые пропустил
```

TanStack Query на клиенте инвалидирует кэш по WebSocket-событиям — UI всегда в актуальном состоянии без polling.

## MVP: домен "Здоровье"

Первая рабочая версия — **один домен**, одна полная вертикаль от регистрации до дебатов агентов.

### Функциональность MVP

| # | Функция | Описание |
|---|---------|----------|
| 1 | Регистрация / логин | JWT + Argon2id, httpOnly cookies |
| 2 | Создание домена "Здоровье" | Только owner видит данные |
| 3 | Telegram-бот | Утреннее напоминание (препараты, ввод веса), вечерний итог дня |
| 4 | Web UI: дашборд | График веса, расписание приёма препаратов, логи агентов |
| 5 | Агент Health Advisor | Отслеживает приём препаратов, анализирует тренд веса, даёт рекомендации |
| 6 | Дебаты | Еженедельный обзор здоровья: аналитик → критик → синтезатор |
| 7 | MD-хранилище | Агент хранит ресерчи, историю рекомендаций в git-versioned MD-файлах |
| 8 | Event log | Каждое взаимодействие зафиксировано в audit log |

### Что MVP доказывает

- Доменная изоляция работает (RLS)
- Агенты работают автономно (cron) и по запросу
- Дебаты дают качественный результат
- Telegram + Web UI — единый поток данных
- MD-память агентов накапливает знания со временем

## Направление развития

> Это не roadmap и не план — это направление, куда продукт может расти после работающего MVP.

**После MVP домена "Здоровье":**

- **Домен "Финансы/Крипто"** — портфель, мониторинг рынка, алерты на события
- **Домен "Работа"** — коллаборация, приглашение коллег, совместные агенты
- **Домен "Фитнес"** — тренировки, телеметрия с умных часов, анализ нагрузки
- **Домен "Личный ассистент"** — календарь, почта, маршруты
- **Масштабирование** до 70+ пользователей

**Технические точки роста:**

| Когда | Что | Зачем |
|-------|-----|-------|
| Auth усложняется | Keycloak | SSO, LDAP, RBAC из коробки |
| pgvector не хватает | Qdrant (dedicated vector DB) | Масштаб и скорость similarity search |
| Нужна оркестрация | Temporal | Auditable workflows, retry, timeout |
| Много пользователей | Горизонтальное масштабирование | Вынести workers, реплики PostgreSQL |
