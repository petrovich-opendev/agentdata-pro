**docker/postgres/init.sql:**
- Create extensions: `uuid-ossp`, `pgvector`, `pg_partman`
- Create role `app_user` with LOGIN and PASSWORD (from env var pattern: `:'app_user_password'` or hardcoded placeholder with comment to use env)
- NOTE: Since PostgreSQL init scripts run as superuser, use `\set` or direct password. The password values come from environment variables `APP_USER_PASSWORD` and are set in docker-compose.yml.
- Actually, use this approach: in init.sql, create roles with placeholder passwords, then in docker-compose.yml pass POSTGRES_INITDB_ARGS or use a shell script wrapper. Simpler: create roles in init.sql with `CREATE ROLE app_user LOGIN PASSWORD 'changeme';` and document in .env.example that production must change these. For Phase 1 dev, this is acceptable.
- Better approach: use `/docker-entrypoint-initdb.d/` shell script that reads env vars:
```sql
-- init.sql
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "vector";
CREATE EXTENSION IF NOT EXISTS "pg_partman";

-- Roles (passwords set via ALTER in shell entrypoint or accept defaults for dev)
DO $$
BEGIN
  IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'app_user') THEN
    CREATE ROLE app_user LOGIN PASSWORD 'app_user_dev_password';
  END IF;
END
$$;

GRANT CONNECT ON DATABASE agentdata TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT USAGE, SELECT ON SEQUENCES TO app_user;
```
